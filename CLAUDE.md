# Vexienne Website Project

## Quick start
Before running any npm/node commands, always run: `source "$HOME/.nvm/nvm.sh"`
Dev server: `cd Website && npm run dev` (runs on http://localhost:4321)
Or use the preview server config in `Website/.claude/launch.json`

## Stack
Astro v6 + Tailwind CSS v4 static site, npm (via nvm), Sharp for image
optimisation, pypdf for PDF extraction. Uses `@tailwindcss/vite` — NOT
`@astrojs/tailwind` (incompatible with Astro 6). Fonts: DM Sans (headings)
and Familjen Grotesk (body) via Google Fonts. Brand palette defined as
Tailwind @theme tokens in `Website/src/styles/global.css`.

## Project structure
- `Website/` — the Astro site
  - `ARCHITECTURE.md` — hosting/SEO/video/email stack decisions and 
    launch checklist (read alongside this file)
  - `src/pages/index.astro` — the complete single-page site (read first 
    for current state)
  - `src/styles/global.css` — brand palette (@theme tokens), fonts, 
    CSS animations
  - `astro.config.mjs` — Astro + Tailwind config
  - `public/images/` — optimised assets (WebP)
  - `dist/` — generated by Astro build. Never edit by hand.
  - `pack.cjs` — script to build the self-contained single HTML file
- `Vexienne_website_preview.html` — generated output from `node pack.cjs`. 
  Never edit by hand. Rebuild after source changes.
- `Vexienne_Brand_Manual_v2.0_DRAFT.pdf` — brand source of truth (45 pages)
- `scratch/` — throwaway files (git-ignored, see rules below)

## File discipline

### Rules
- When exploring, prototyping, or writing one-off scripts to check 
  something, put them in `/scratch` (at repo root). Do not scatter them 
  across the repo root or `Website/src`.
- Delete files in `/scratch` as soon as they've served their purpose, 
  not only at task end. If you create a script to check something and 
  the check is done, delete it in the same step.
- Never edit `Website/dist/` or `Vexienne_website_preview.html` directly. 
  Change `Website/src/` and rebuild.
- `/scratch` is only for throwaway experiments. There is no existing 
  test suite; if tests are introduced, they go in a proper `tests/` 
  location, not in `/scratch`.
- Don't create new top-level folders without asking.

### End-of-task checkpoint
Before telling me a task is complete, always produce a short report with:

1. Files created or modified — full list, grouped by folder.
2. `/scratch` status — list what's still there and why, or confirm empty.
3. Cleanup suggestion — if anything in `/scratch` is no longer needed, 
   say so and offer to delete it before closing out.
4. Rebuild reminder — if `Website/src/` changed, remind me to run 
   `cd Website && node pack.cjs` to refresh the preview HTML.

Do not skip this checkpoint, even for small tasks. If the task was 
trivial and nothing was created, a one-line confirmation is fine.

### Hard backstop
There is an `npm run clean:scratch` command that wipes `/scratch`. 
I may run it between tasks. Do not store anything in `/scratch` that 
you expect to persist across tasks — if something needs to survive, 
it belongs in `Website/src/`, a proper tests folder, or `/docs`, not 
`/scratch`.
